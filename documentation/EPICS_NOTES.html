Introduction

This support module uses a LEEP driver to communicate with NASA ACQ nodes and provides an example IOC, and some Engineering CSS screens.  Most of the records mentioned in the following sections have names beginning with $(P)$(NODE) macros.  Typically the P macro is set to 'FDAS:' and the NODE macro to the acquisition node number (01 through 32) followed by a colon.

When acquisition is enabled acquisition activity is delayed until the middle of the next sample interval.  This ensures that there are no race conditions between nodes beginning to fill their acquisition buffers.  When acquisition is disabled acquisition activity continues until the currently-active packet has been filled and transmitted.

Per-Node Status

The time at which the firmware and software were built is contained in the per-node stringin records firmwareBuildDate and
softwareBuildDate, respectively.

The IPMI serial number of the Quartz FMC card is contained in the per-node longin record Quartz:serno and the part number (PCB revision) is in Quartz:partno.  A Quartz:partno of 0 indicates that the IPMI EEPROM on the Quartz card is invalid in some way.  A Quartz:partno less than 0 indicates a mismatch between the firmware and hardware.

The per-node longin record ACQ:seqno contains the least-significant 32 bits of the most recent high-speed data packet sequence number.

The per-node aai record Quartz:ADC:headers contains a snapshot of the Quartz AD7768 analog to digital converter header bytes.  The order of the values in the array is that of the front panel inputs (the first value in the array is the status of the AD7768 channel for front panel input 1).  Because of the way the inputs are connected this order does not match that of the channels in the AD7768 chips.  The SCAN field is 'Passive' so the values are updated only when a client explicitly requests record processing.

The per-node longin record Quartz:ADC:statuses contains the Quartz AD7768 analog to digital converter status register values.  Each byte of the 32-bit word is the value of register 0x09 for one AD7768.  The most-significant byte corresponds to ADC 0 and the least-significant byte to ADC 3.  The CHIP_ERROR status is bit 3 (0x08) of each byte.  Try to process the record sparingly, if at all, during acquisition since it increases the noise.  The SCAN field is 'Passive' so the values are updated only when a client explicitly requests record processing.

The history of the Quartz analog to digital converter DRDY lines is captured in the longin record DRDY:history.  Each nibble, shown as a hexadecimal value, holds the status of the four DRDY lines with an 8 ns sampling clock.  The least significant bit of each nibble corresponds to the first ADC.  The least significant nibble is the value of the lines at the first rising edge of any of the lines after all lines have been low.  Sampling stops when all four lines are high.  The following table describes the conditions associated with some possible values where 'x' indicates a "don't care" value and 'y' indicates a value other then 'F'.  The most significant three bits of this record reflect the current state of the state machine which acquires the history.

Calibration

Calibration values are read or written using 32-element, FTVL=LONG, aai/aao records.  Values are applied to the ADC chips immediately when the aao records are processed but are not stored in non-volatile memory until the CAL:write record mentioned below is processed.

  • CAL:offset/CAL:offsetRbk – per-channel calibration offsets in units of ADC counts.  Range is ±1000000.
  • CAL:gain/CAL:gainRbk – per-channel calibration gain adjustments in units of parts per million.  Range is ±50000.

There are four other scalar records associated with the calibration

  • CAL:date/CAL:dateRbk –  the time associated with the calibration values, in POSIX seconds, treated as an unsigned 32-bit value.
  • CAL:status – MBBI record indicating the status of the calibration read/write operations.
  • CAL:write – BO record that writes the calibration values (offset, gain, date) to non-volatile memory when processed.

On IOC startup the output records are disabled (the DISA and DISV fields have the same value) to reduce the chance of accidentally changing the calibration values.

Machine Protection

The following table describes the records associated with the machine protection operations handled locally by each node.  The value oo is the machine protection mitigation output number, ranging from 01 to 04.

Digital inputs and outputs are optically isolated.  An open or undriven input appears as a logic 0.  The transistor of an output opto-isolator is OFF when the output is in the 'tripped' state.  On power-up all outputs are in the 'tripped' state.  On the event generator node the outputs are driven by the combination of local tripped status from all nodes as described below by the MPS:required and  MPS:tripped records.  On all other nodes the outputs are driven by the local 'tripped' as described below by the MPS:status:oo records.

Internal Records

Records with names ending in an underscore are intended for internal use by the IOC and should not be written by any other clients.
